# 局域网聊天室（CLI）—— 完整架构设计文档

## 1. 项目概述

本项目是一个基于 TCP 的局域网聊天室 CLI 程序，采用客户端-服务器中心化架构，并提供全屏终端界面。支持群聊、私聊、频道/房间、文件传输、用户在线列表、改名等完整功能，并通过严格的密码安全机制与令牌鉴权保障通信安全。系统具备心跳检测、超时断开和断线重连能力，并使用 SQLite 持久化消息历史与用户数据。架构上预留了去中心化（P2P）扩展接口。

## 2. 技术选型

| 技术项   | 选择                                                         |
| -------- | ------------------------------------------------------------ |
| 运行时   | Node.js 18+ LTS + TypeScript 5.x                             |
| 通信底层 | TCP（Node `tls` 模块）                                       |
| 应用协议 | 自定义二进制协议，控制/聊天负载使用 JSON，文件传输支持二进制块 |
| 信道加密 | TLS 1.2/1.3 + 自签名 X.509 证书 + 客户端证书指纹固定         |

| TUI 框架 | `blessed`（全屏彩色界面）                                    |
| 文件传输 | 服务器中转，支持断点续传，单文件最大 500 MB                  |
| 日志     | 结构化日志输出到文件与控制台，采用 `winston` 库              |

## 4. 通信协议详细设计

### 4.1 符号约定

- C：客户端 (Client)
- S：服务器 (Server)
- 多字节整数均使用网络字节序（Big Endian）

### 4.2 二进制帧格式

每条消息由一个固定头部和可变长度负载组成：

```
| 4 字节 总长度 (Big Endian) | 1 字节 消息类型 | N 字节 Payload |
```

- **总长度**：包含该字段自身（4 字节）、消息类型字段（1 字节）以及 Payload 长度。最小有效长度为 5（即 Payload 长度为 0）。
- **消息类型**：1 字节无符号整数，标识消息类别。有效范围 0x00 ~ 0xFF。
- **Payload**：对于控制信令、聊天消息等，采用 UTF-8 编码的 JSON 字符串；对于文件数据块，则直接填充二进制内容。

### 4.3 消息类型定义

#### 4.3.1 系统与控制类

| 类型码 | 名称                | 方向 | 说明                                                         |
| ------ | ------------------- | ---- | ------------------------------------------------------------ |
| `0x01` | `LOGIN_REQUEST`     | C→S  | 登录请求，Payload: `{ nickname: string, password: string }`  |
| `0x02` | `LOGIN_RESPONSE`    | S→C  | 登录结果，Payload: `{ success: boolean, token?: string, error?: string }` |
| `0x03` | `REGISTER_REQUEST`  | C→S  | 注册请求，Payload 同 `LOGIN_REQUEST`                         |
| `0x04` | `REGISTER_RESPONSE` | S→C  | 注册结果，Payload: `{ success: boolean, error?: string }`    |
| `0x05` | `HEARTBEAT`         | S→C  | 服务端心跳包，无 Payload                                     |
| `0x06` | `HEARTBEAT_ACK`     | C→S  | 客户端心跳应答，无 Payload                                   |
| `0x07` | `DISCONNECT`        | 双向 | 主动断开通知，Payload: `{ reason?: string }`                 |
| `0x08` | `ERROR`             | S→C  | 错误信息，Payload: `{ code: string, message: string }`       |

#### 4.3.2 房间与用户

| 类型码 | 名称                   | 方向 | 说明                                                         |
| ------ | ---------------------- | ---- | ------------------------------------------------------------ |
| `0x10` | `ROOM_LIST`            | S→C  | 所有房间列表，Payload: `{ rooms: Array<{ id: number, name: string, memberCount: number }> }` |
| `0x11` | `USER_LIST`            | S→C  | 当前房间用户列表，Payload: `{ room: string, users: string[] }` |
| `0x12` | `ROOM_JOIN`            | C→S  | 加入房间请求，Payload: `{ roomName: string }`                |
| `0x13` | `ROOM_JOIN_RESPONSE`   | S→C  | 加入结果，Payload: `{ room: string, success: boolean, error?: string }` |
| `0x14` | `ROOM_LEAVE`           | C→S  | 离开房间请求，Payload: `{ roomName: string }`                |
| `0x15` | `USER_JOINED`          | S→C  | 广播用户加入房间，Payload: `{ nickname: string, room: string }` |
| `0x16` | `USER_LEFT`            | S→C  | 广播用户离开或断开连接，Payload: `{ nickname: string, room: string }` |
| `0x17` | `NICK_CHANGE`          | C→S  | 改名请求，Payload: `{ newNickname: string, token: string }`  |
| `0x18` | `NICK_CHANGE_RESPONSE` | S→C  | 改名结果，Payload: `{ success: boolean, newNickname?: string, error?: string }` |

#### 4.3.3 聊天消息

| 类型码 | 名称               | 方向      | 说明                                                         |
| ------ | ------------------ | --------- | ------------------------------------------------------------ |
| `0x20` | `CHAT_ROOM`        | C→S / S→C | 房间消息，Payload: `{ room: string, text: string, timestamp: string, sender: string }` |
| `0x21` | `CHAT_PRIVATE`     | C→S / S→C | 私聊消息，Payload: `{ target: string, text: string, timestamp: string, sender: string }` |
| `0x22` | `CHAT_SYSTEM`      | S→C       | 系统通知（加入、离开、改名等），Payload: `{ text: string, timestamp: string }` |
| `0x23` | `HISTORY_REQUEST`  | C→S       | 请求历史消息，Payload: `{ room?: string, type?: 'room'\|'private', count?: number }` |
| `0x24` | `HISTORY_RESPONSE` | S→C       | 返回历史消息数组，Payload: `{ messages: Array<{ sender: string, content: string, timestamp: string }> }` |

#### 4.3.4 文件传输

| 类型码 | 名称            | 方向       | 说明                                                         |
| ------ | --------------- | ---------- | ------------------------------------------------------------ |
| `0x30` | `FILE_REQUEST`  | C→S        | 发送文件请求，Payload: `{ fileName: string, fileSize: number, targetUser?: string, room?: string }` |
| `0x31` | `FILE_RESPONSE` | S→C 或 C→S | 接受/拒绝/续传信息，Payload: `{ transferId: string, accepted: boolean, nextChunkIndex?: number, reason?: string }` |
| `0x32` | `FILE_CHUNK`    | C→S / S→C  | 文件数据块，二进制布局：`transferId(32字节ASCII字符串) + chunkIndex(4字节无符号整数) + 数据` |
| `0x33` | `FILE_END`      | 双向       | 传输结束通知，Payload: `{ transferId: string, status: 'success'\|'aborted', reason?: string }` |
| `0x34` | `FILE_PROGRESS` | S→C        | 传输进度更新，Payload: `{ transferId: string, receivedBytes: number, totalBytes: number }` |

### 4.4 文件传输与断点续传流程

1. **发起请求**：发送方构造 `FILE_REQUEST`，指明目标用户（私聊）或房间（群文件）。服务器验证目标存在且文件大小不超过 500 MB，生成唯一 `transferId` 并通知接收方。  
2. **协商**：接收方收到请求后，可在 TUI 中确认或拒绝。若接受，检查本地或服务器是否已有未完成的传输记录，回复 `FILE_RESPONSE`，其中 `nextChunkIndex` 表示期望的下一个块序号（0 表示全新传输）。  
3. **数据传输**：发送方将文件按 **64 KB** 固定大小分块（最后一块可能不足 64 KB）。从 `nextChunkIndex` 开始，依次发送 `FILE_CHUNK`。服务器收到块后，写入临时文件，同时转发给接收方，并记录已接收的块位图。  
4. **进度反馈**：服务器每收到 10 个块或每 1 秒，向发送方和接收方发送 `FILE_PROGRESS`。  
5. **完成与清理**：所有块发送完毕，发送方发送 `FILE_END`（status: success）。服务器校验文件大小一致性，将临时文件重命名为正式存储文件，并在数据库中记录文件元信息。若传输中断，服务器保留已接收的块及位图，以供后续 `FILE_RESPONSE` 续传。临时文件 24 小时内未被续传则自动清理。

### 4.5 鉴权与令牌携带

除 `LOGIN_REQUEST`、`REGISTER_REQUEST` 和 `HEARTBEAT` 外，所有客户端请求的 JSON Payload 中必须包含 `"token": "<JWT>"` 字段。服务端在执行路由前统一验证令牌有效性。

## 5. 安全设计

### 5.1 信道加密

- 服务端启动时加载自签名 X.509 证书（RSA 2048 位，有效期 10 年），强制使用 TLS 1.2 或更高版本。
- 客户端首次连接时，展示服务端证书的 SHA-256 指纹（十六进制格式），要求用户人工比对（或直接确认）。确认后指纹存入本地文件 `~/.lanchat/known_hosts`。
- 后续连接时自动比对指纹，若不一致则拒绝连接并显示警告。

### 5.2 密码哈希

- 服务端使用 Argon2id 算法对密码进行哈希，参数：`timeCost=4`、`memoryCost=65536` (KiB)、`parallelism=2`、`hashLength=32` 字节。盐值由算法随机生成并嵌入结果字符串。
- 数据库中仅存储该 Argon2id 哈希字符串，永远不存储明文密码。
- 验证时使用 Argon2id 的验证函数比对明文与哈希。

### 5.3 JWT 令牌

- 登录成功后，服务端签发 JWT，Payload 为：
  ```json
  {
    "sub": "<userId>",
    "nickname": "<nickname>",
    "iat": <issuedAt>,
    "exp": <expiresAt>
  }
  ```
- 签名算法为 `HS256`（HMAC-SHA256）。签名密钥通过环境变量 `JWT_SECRET` 注入，要求长度不低于 32 字符的随机字符串。
- 令牌有效期为签发后 24 小时。客户端必须安全存储令牌，并在每次需要鉴权的请求中携带。
- 密码修改：客户端发送当前密码与新密码（均在 TLS 保护下），服务端验证旧密码后更新 Argon2id 哈希，并要求客户端重新登录（旧令牌失效）。

### 5.4 输入校验

- 昵称：长度 3~20 字符，仅允许大小写字母、数字、下划线，首字符必须为字母。
- 房间名：长度 1~30 字符，允许字母、数字、连字符 `-`、下划线 `_`，必须带有 `#` 前缀（如 `#general`）。
- 消息文本：最长 5000 字符。
- 文件大小：1 B ≤ 大小 ≤ 500 MB（0 字节文件不接受传输）。
- 所有数据库操作均使用参数化查询，防范 SQL 注入。

### 5.5 注册策略

系统采用开放注册模式：任何知道服务器地址并完成 TLS 握手校验的用户，均可通过 `REGISTER_REQUEST` 创建新账号。注册时昵称必须全局唯一。

## 6. 数据库设计

数据库使用 SQLite 3，文件路径 `data/lanchat.db`，开启 WAL 模式以提升并发读取性能。

### 6.1 表结构

```sql
-- 用户表
CREATE TABLE users (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    nickname   TEXT    UNIQUE NOT NULL,
    password   TEXT    NOT NULL,          -- Argon2id 哈希结果
    created_at TEXT    DEFAULT (datetime('now'))
);

-- 房间表
CREATE TABLE rooms (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    name       TEXT    UNIQUE NOT NULL,   -- 含 '#' 前缀，如 '#general'
    created_by INTEGER REFERENCES users(id),
    is_default INTEGER DEFAULT 0,        -- 1 表示默认房间，不可删除
    created_at TEXT    DEFAULT (datetime('now'))
);

-- 房间消息（群聊历史）
CREATE TABLE messages (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    room_id    INTEGER REFERENCES rooms(id) NOT NULL,
    sender_id  INTEGER REFERENCES users(id) NOT NULL,
    content    TEXT    NOT NULL,
    type       TEXT    DEFAULT 'chat',   -- 'chat' 或 'system'
    timestamp  TEXT    DEFAULT (datetime('now'))
);

-- 私聊消息历史
CREATE TABLE private_messages (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    sender_id   INTEGER REFERENCES users(id) NOT NULL,
    receiver_id INTEGER REFERENCES users(id) NOT NULL,
    content     TEXT    NOT NULL,
    timestamp   TEXT    DEFAULT (datetime('now'))
);

-- 文件元数据
CREATE TABLE files (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    transfer_id TEXT    UNIQUE NOT NULL,
    sender_id   INTEGER REFERENCES users(id),
    receiver_id INTEGER,                 -- 私聊时为接收者用户ID，群文件时为 NULL
    room_id     INTEGER REFERENCES rooms(id),
    file_name   TEXT    NOT NULL,
    file_size   INTEGER NOT NULL,
    stored_path TEXT,                    -- 服务器文件相对路径
    timestamp   TEXT    DEFAULT (datetime('now'))
);
```

### 6.2 索引

```sql
CREATE INDEX idx_messages_room_time ON messages (room_id, timestamp);
CREATE INDEX idx_private_messages_users_time ON private_messages (sender_id, receiver_id, timestamp);
CREATE INDEX idx_users_nickname ON users (nickname);
```

### 6.3 默认数据

应用首次启动时，自动创建默认房间：
```sql
INSERT INTO rooms (name, is_default) VALUES ('#general', 1);
```

### 6.4 历史消息策略

- 用户加入房间时，服务端自动推送最近 50 条房间消息。
- 私聊窗口打开时，客户端可主动请求最近 50 条私聊历史。
- 客户端可通过 `/history [n]` 命令请求更多历史记录，默认 n=50，最大不超过 200。

## 7. 服务端架构

### 7.1 核心模块职责

| 模块                  | 职责描述                                                     |
| --------------------- | ------------------------------------------------------------ |
| `TlsServer`           | 创建并管理 `tls.Server`，监听指定端口，接收新连接并创建 `ClientConnection`。 |
| `ClientConnection`    | 封装单个 TLS socket，维护接收缓冲区，按帧边界解包。管理连接状态，启动心跳计时器，并将解码后的消息交给 `MessageRouter`。 |
| `AuthManager`         | 处理注册与登录请求，执行 Argon2id 哈希与验证，签发和验证 JWT。 |
| `UserManager`         | 维护在线用户映射（socketId → { userId, nickname, activeRoom }），确保昵称在线唯一，触发用户上线/下线事件。 |
| `RoomManager`         | 管理房间列表及每个房间的成员集合。处理创建、加入、离开，并广播成员变更。默认房间不可删除且新用户自动加入。 |
| `MessageRouter`       | 作为消息分发中枢：首先调用 `AuthManager.verifyToken()` 进行鉴权（除豁免类型），然后按消息类型路由到 `ChatService` 或 `FileService`，最后通过 `ClientConnection.send()` 完成单播或广播。 |
| `FileTransferHandler` | 管理文件传输会话：为每个 `transferId` 维护进度位图、临时文件句柄，处理数据块写入、转发、完成通知和续传查询。 |
| `HeartbeatService`    | 定时（15 秒）向所有已认证连接发送 `HEARTBEAT`；对每个连接设置 10 秒超时计时器，未收到 `HEARTBEAT_ACK` 则主动断开并清理资源。 |
| `Database`            | 负责 SQLite 连接管理、执行 migrations，并提供事务支持。      |
| `Repositories`        | 对各个数据表的 CRUD 封装，提供异步或同步接口供 Service 层调用。 |
| `Services`            | 业务逻辑层，协调 Repository 与在线状态管理器，完成具体操作并返回结果。 |

### 7.2 连接生命周期

```
[新连接] → TLS 握手 → 连接建立（状态：unauthenticated）
    │
    ├─ 收到 LOGIN_REQUEST / REGISTER_REQUEST → AuthManager 处理
    │     ├─ 成功：状态变更为 authenticated，返回 JWT，加入 UserManager，
    │     │        自动加入默认房间（#general），启动心跳检测。
    │     └─ 失败：返回错误消息，保持 unauthenticated。
    │
    ├─ 收到其他消息 → 检查令牌（若未携带有效令牌则返回 AUTH_ERROR）。
    │
    └─ 心跳超时 / 收到 DISCONNECT / socket 关闭事件
          → 从 UserManager 移除，从所有房间移除，
          → 广播 USER_LEFT 给相关房间，
          → 终止相关文件传输会话，释放连接资源。
```

### 7.3 消息路由详例（房间聊天）

1. `ClientConnection` 解析出一个 `CHAT_ROOM` 消息，调用 `MessageRouter.route(socketId, message)`。  
2. `MessageRouter` 从 Payload 中提取 `token`，调用 `AuthManager.verifyToken(token)` 得到 `userId` 和 `nickname`。  
3. 从 `UserManager` 获取该连接的 `activeRoom`（即用户当前所在房间）。  
4. 调用 `ChatService.sendRoomMessage(room, userId, nickname, text)`：  
   - 将消息写入 `messages` 表。  
   - 构造完整 Payload（含 sender、timestamp）。  
5. `MessageRouter` 通过 `RoomManager.getMembers(room)` 获取该房间所有成员的 socketId 列表。  
6. 遍历列表，通过 `ClientConnection.getById()` 获取对应连接对象，调用其 `send()` 方法发送消息，**排除发送者自身**。

### 7.4 心跳与超时

- 服务端心跳间隔：15 秒。
- 客户端心跳应答超时：10 秒。
- 客户端无数据超时判定：30 秒内未收到任何来自服务端的数据（包括心跳），视为连接中断，触发重连流程。
- 上述超时值定义为 `constants.ts` 中的可配置常量。

### 7.5 并发与流控

- 服务端对每个连接的消息发送采用顺序发送，保证消息顺序。使用 Node.js Socket 的背压机制（`drain` 事件）避免缓冲区溢出。
- 文件传输使用流式读取，不将大文件完全加载到内存。服务器转发时采用被动模式（接收一块、写入、转发一块），保证内存占用可控。

## 8. 客户端架构

### 8.1 连接状态机

客户端连接状态定义：

- `Disconnected`：初始状态，未连接或已断开。
- `Connecting`：正在建立 TCP 连接并进行 TLS 握手。
- `Connected`：安全连接已建立，但尚未登录认证。
- `Authenticated`：已成功登录，持有有效 JWT，可进行聊天操作。
- `Reconnecting`：连接意外断开，正在执行重连策略。

状态迁移规则：

| 当前状态      | 事件 / 触发条件              | 目标状态      |
| ------------- | ---------------------------- | ------------- |
| Disconnected  | 用户发起连接                 | Connecting    |
| Connecting    | TLS 握手成功                 | Connected     |
| Connecting    | 连接失败                     | Disconnected  |
| Connected     | 收到 `LOGIN_RESPONSE` 且成功 | Authenticated |
| Connected     | 登录失败或主动断开           | Disconnected  |
| Authenticated | 检测到连接断开（超时/错误）  | Reconnecting  |
| Authenticated | 用户主动退出（/quit）        | Disconnected  |
| Reconnecting  | 重连成功并自动重认证成功     | Authenticated |
| Reconnecting  | 重试次数耗尽或用户主动取消   | Disconnected  |
| 任何状态      | 程序退出                     | 程序终止      |

重连策略：指数退避，初始重试间隔 1 秒，每次翻倍，最大间隔 16 秒，最多重试 5 次。重连时自动发送存储的 JWT 进行身份恢复。若令牌已过期（服务端返回认证错误），客户端退回 `Disconnected` 并提示用户重新登录。

## 9. 去中心化预留

为支持未来向 P2P 架构过渡，客户端网络层抽象为 `Transport` 接口：

```typescript
interface Transport {
  connect(host: string, port: number): Promise<void>;
  send(data: Buffer): void;
  onMessage(callback: (data: Buffer) => void): void;
  onClose(callback: () => void): void;
  close(): void;
  getRemoteFingerprint?(): string;
}
```

当前实现 `TlsTransport` 基于 TLS/TCP。未来只需实现 `P2pTransport`（例如使用 `hyperswarm` 或 `libp2p`），并保持接口不变，`ChatClient` 及其上层逻辑无需修改。服务端路由模块也可抽象为 `MessageHub`，以便在对等网络中作为超级节点复用。具体 P2P 发现、NAT 穿透等问题在实施新模块时解决。

## 10. 错误处理与日志

### 10.1 错误码体系

服务端定义标准错误码，通过 `ERROR` 消息返回给客户端：

| 错误码            | 含义              |
| ----------------- | ----------------- |
| `AUTH_FAILED`     | 登录/令牌验证失败 |
| `NICK_TAKEN`      | 昵称已被占用      |
| `INVALID_MESSAGE` | 消息格式不合法    |
| `ROOM_NOT_FOUND`  | 房间不存在        |
| `TARGET_OFFLINE`  | 目标用户不在线    |
| `FILE_TOO_LARGE`  | 文件超过大小限制  |
| `INTERNAL_ERROR`  | 服务器内部错误    |

客户端根据错误码显示对应的中文提示信息。

### 10.2 日志

- 服务端使用 `winston` 记录结构化日志，输出 JSON 格式。日志级别通过环境变量 `LOG_LEVEL` 控制（默认 `info`）。同时输出到标准输出和 `logs/server.log`。
- 关键事件记录：连接建立/断开、登录成功/失败、文件传输开始/结束、异常堆栈。
- 客户端仅在本地文件记录错误日志（路径 `~/.lanchat/error.log`），方便问题定位。

### 10.3 全局异常处理

服务端和客户端均监听 `uncaughtException` 和 `unhandledRejection` 事件，记录异常并尝试优雅关闭。服务端不因单个连接异常而崩溃。

## 11. 部署与运维

### 11.1 环境变量

项目根目录放置 `.env.example` 文件，实际环境变量通过 `.env` 管理：

```
# 服务端配置
PORT=9527
JWT_SECRET=<至少32字符的随机字符串>

DEFAULT_PORT=9527
```

### 11.2 证书生成

使用 OpenSSL 生成自签名证书，有效期 10 年：
```bash
openssl req -x509 -nodes -days 3650 -newkey rsa:2048 \
  -keyout certs/server.key \
  -out certs/server.crt \
  -subj "/CN=LanChat"
```

### 11.3 启动顺序

1. 安装依赖：`npm install`
2. 编译 TypeScript：`npm run build`
3. 启动服务端：`npm run start:server`
4. 启动客户端：`npm run start:client` 或全局安装后直接运行 `lanchat`

### 11.4 安全注意事项

- `.env` 和 `certs/server.key` 必须加入 `.gitignore`，严禁提交到版本库。
- 服务端运行目录应设置合适的文件权限，防止 `data/` 目录和证书被非授权访问。
- 生产环境部署时，宜使用内部 CA 签发的正式证书，简化客户端指纹确认流程。

## 12. 非功能需求

- **性能**：在常规硬件上支持至少 200 个并发连接。消息收发延迟低于 100ms。文件传输对聊天消息无阻塞影响。
- **可靠性**：自动重连与心跳机制保障网络波动下的可用性。SQLite WAL 模式避免读写冲突，数据库操作失败时返回明确错误而非崩溃。
- **可维护性**：严格遵循模块化设计，各模块职责单一，接口清晰。TypeScript 强类型约束降低协作成本。代码注释覆盖公共 API 和复杂算法。
